Generate an implementation task list for a BearLink spec file.

Arguments: `$ARGUMENTS`

Parse the arguments as follows:

- First token: path to the spec file (required), e.g. `spec/my-feature.md`

---

## Step 1 — Read the spec

Read the spec file provided in the arguments. Understand:

- The Goal
- Every requirement in the Requirements section (functional and non-functional)
- The service(s) affected (`**Service(s):**` front-matter)
- Whether the spec introduces a new service or modifies existing ones
- The Docs to Update section (openapi.yaml, asyncapi.yaml)

Also read `spec/done/auth-service-outbox.md` and `spec/done/audit-service.md` as
reference examples of well-formed task lists.

## Step 2 — Check preconditions

- If the spec already has a non-empty `## Tasks` section with at least one task, report
  that tasks already exist and stop. Do not overwrite existing tasks.
- If the spec status is `draft`, warn that tasks should only be generated for approved
  specs, then stop.

## Step 3 — Plan the tasks

Group the requirements into a small number of coherent, independently-committable tasks.
Apply these rules:

**Ordering — always follow this sequence when applicable:**

1. Schema / migration changes (Prisma model, migration SQL) — always first when a DB
   change is needed, because all other tasks depend on the model being defined.
2. Core implementation (controllers, services, routes) — the business logic.
3. Infrastructure changes (docker-compose, Kubernetes, env vars) — if needed.
4. Tests and documentation (vitest test files, openapi.yaml, asyncapi.yaml) — always
   last, because tests verify the implementation.

If the spec introduces a new service, add a dedicated scaffold task before the
implementation tasks (package.json, Dockerfile, index.js, app.js, routes/index.js,
CLAUDE.md).

**Sizing — each task should:**

- Be completable in one focused session without switching context mid-task.
- Touch a coherent set of files (e.g. "controller + route" is one task;
  "schema + migration" is one task; "tests + docs" is one task).
- Not be so large that it mixes unrelated concerns (e.g. do not combine schema changes
  with test updates in the same task).
- Reference the requirement numbers it satisfies (e.g. `(R1-R3, R8)`).

**Typical task count:** 2-5 tasks. Avoid splitting trivially small specs into more
tasks than necessary.

## Step 4 — Write the tasks

Append a `## Tasks` section to the end of the spec file (after the last existing
section). Use this format exactly:

```
## Tasks

<!-- Generated by Claude from the requirements above. Do not write these manually. -->

### Task N -- <Short descriptive title> (<requirement refs>) [ ]

File: `path/to/file.js`              <!-- use "File:" for a single file -->
Files: `path/to/a.js`,               <!-- use "Files:" for multiple files -->
       `path/to/b.js`

- <Concrete action bullet. Name the function, method, or model being changed.>
- <Another concrete action. Reference field names, endpoint paths, or event types
  where known.>
- <...>
```

Rules for task content:

- Each bullet must be a concrete action, not a vague goal. "Add `OutboxEvent` model
  with fields: id (Int autoincrement PK), ..." is good. "Set up the database" is not.
- Name exact file paths relative to the repo root.
- When a file is new, append `(new)` after the path.
- Reference field names, function signatures, endpoint paths, payload shapes, and
  environment variable names wherever the spec specifies them.
- When a task involves both a callback-form and array-form `$transaction` (outbox
  pattern), spell out which handlers use which form.
- For test tasks: list each test scenario as a bullet, not just "write tests".
- Do not include implementation details that the spec does not specify — do not invent
  requirements.

## Step 5 — Verify and write

Re-read the generated tasks against the spec requirements. Check:

- Every functional requirement (R1, R2, ...) is covered by at least one task bullet.
- No task bullet introduces something not in the spec.
- The Docs to Update items (openapi.yaml, asyncapi.yaml) are covered in the final task
  if they require changes, or explicitly noted as "no update needed" if they do not.
- File paths are plausible given the service(s) named in the spec front-matter.

Write the Tasks section to the spec file. Do not change any other part of the spec
(do not modify status, requirements, acceptance criteria, or any other section).

Report a brief summary of the tasks generated.

---

## Rules

- Never modify any part of the spec except appending the `## Tasks` section.
- Never change the spec status.
- Do not generate tasks for a spec that already has them.
- Do not generate tasks for a `draft` spec.
- Use plain ASCII only — no Unicode box-drawing characters.
- Keep task titles concise (under 60 characters).
