# Plan Enforcement

> **Status:** draft
> **Service(s):** url-service, analytics-service, shared
> **Priority:** high

## Goal

Gate all premium features behind plan checks so that only users with an active
paid subscription can use them. Enforcement happens at the API layer in each
backend service using the plan value already present in the JWT.

## Background

After `spec/plan-model.md` is implemented, `req.user.plan` is available in every
service. This spec wires that value to feature guards. No new network calls are
required at request time — enforcement is purely JWT-based.

See `spec/roadmap/premium-plans-roadmap.md` for the feature-gate matrix.

## Requirements

### Functional

- **R1:** The shared middleware module shall export a `requirePlan(plan)` helper
  that returns a 403 `{ error: "plan_required", requiredPlan: "pro" }` if
  `req.user.plan` does not meet the required level.
- **R2:** Plan levels shall be ordered: `free` < `pro` < `enterprise`. A user
  on `pro` satisfies a `requirePlan("pro")` check; a user on `enterprise` satisfies
  all checks.
- **R3:** The url-service shall apply `requirePlan("pro")` to the following features:
  - Custom alias (field `customAlias` in `POST /urls` and `PUT /urls/:id`)
  - Link expiry (`expiresAt`)
  - Password protection (`password`)
  - Tags (`tags`)
  - UTM parameters (`utmParams`)
  - Redirect type selection (`redirectType` -- free users always get 302)
  - QR code endpoint (`GET /:shortId/qr`)
  - Bulk create (`POST /urls/bulk`)
  - Signed URL creation (`POST /urls/:id/sign` and `requireSignature` flag)
- **R4:** The analytics-service shall apply `requirePlan("pro")` to
  `GET /analytics/urls/:shortId/clicks`.
- **R5:** The url-service shall enforce a monthly URL creation quota per user:
  25 URLs/month for `free`, 500 URLs/month for `pro`. Exceeding the quota returns
  429 `{ error: "quota_exceeded", limit: 25, plan: "free" }`.
- **R6:** Quota shall be tracked in the url-service database (a new
  `MonthlyQuota` model: `userId`, `yearMonth` YYYYMM, `count`); no Redis required.
- **R7:** Existing URLs and analytics events shall remain accessible regardless of
  plan changes (no retroactive removal).

### Non-Functional

- **R8:** `requirePlan` shall be a pure function with no I/O — it reads only from
  `req.user` so that it adds zero latency to requests.
- **R9:** All enforcement points shall be covered by integration tests that verify
  both 403 (wrong plan) and 200 (correct plan) responses.
- **R10:** The 403 response body shall always include `requiredPlan` so clients can
  prompt the user to upgrade.

## Acceptance Criteria

- [ ] A free user calling `POST /urls` with `customAlias` set receives 403 with `error: "plan_required"`.
- [ ] A pro user calling `POST /urls` with `customAlias` set succeeds (201).
- [ ] A free user calling `GET /:shortId/qr` receives 403.
- [ ] A free user calling `POST /urls/bulk` receives 403.
- [ ] A free user who has created 25 URLs this month receives 429 on the 26th attempt.
- [ ] A pro user who has created 500 URLs this month receives 429 on the 501st attempt.
- [ ] A free user calling `GET /analytics/urls/:shortId/clicks` receives 403.
- [ ] Existing short links created before plan enforcement still redirect correctly.
- [ ] `requirePlan("pro")` in shared middleware passes unit tests for all plan levels.

## Out of Scope

- Stripe or payment flows.
- Downgrade handling (existing premium links are NOT disabled on downgrade -- left for a future spec).
- Admin-only endpoints (unaffected by plan).
- Preview-service and notification-service (no plan-gated features).
