# Billing UI

> **Status:** approved
> **Service(s):** web-ui
> **Priority:** medium

## Goal

Give users visibility into their current plan and a clear path to upgrade,
downgrade, or manage their subscription -- all through the existing Next.js
frontend.

## Background

After Parts 1-3 of the premium-plans initiative, the backend can accept payments
and enforce plan limits. This spec surfaces that to users: a pricing page,
an upgrade CTA wherever a plan gate is hit, a plan badge in the header, and a
billing settings page.

## Requirements

### Functional

- **R1:** The app shall display a `/pricing` page (public, no auth required) that
  lists Free and Pro plan features and prices side by side with a clear "Upgrade"
  CTA.
- **R2:** The navigation header shall display a plan badge (e.g. "Free" or "Pro")
  for logged-in users, sourced from the JWT payload.
- **R3:** When an API call returns 403 with `error: "plan_required"`, the frontend
  shall display an upgrade prompt (modal or inline banner) with a link to `/pricing`
  or directly to the checkout flow.
- **R4:** When an API call returns 429 with `error: "quota_exceeded"`, the frontend
  shall display a quota message and an upgrade CTA.
- **R5:** A `/settings/billing` page (authenticated) shall show:
  - Current plan and status
  - `currentPeriodEnd` formatted as a human-readable date (for pro users)
  - "Manage subscription" button (calls `POST /billing/portal`, redirects to Stripe
    Customer Portal)
  - "Upgrade to Pro" button (calls `POST /billing/checkout`, redirects to Stripe
    Checkout) for free users
- **R6:** After returning from Stripe Checkout (success URL), the frontend shall
  display a success message and reflect the updated plan (requires a JWT refresh).
- **R7:** The "Create URL" form shall hide premium fields (custom alias, expiry,
  password, tags, UTM, redirect type) for free users, with a "Pro feature" tooltip
  and upgrade link rather than allowing the user to fill them in and get a 403.

### Non-Functional

- **R8:** No new UI component libraries shall be introduced -- use the existing
  shadcn/ui component set.
- **R9:** All new pages shall follow the existing Next.js App Router conventions
  used in the project.
- **R10:** The pricing page shall be statically renderable (no server-side user
  data required).

## Acceptance Criteria

- [ ] `/pricing` renders for unauthenticated users and shows Free vs. Pro comparison.
- [ ] A logged-in free user sees a "Free" badge in the header.
- [ ] A logged-in pro user sees a "Pro" badge in the header.
- [ ] Clicking "Upgrade to Pro" on `/settings/billing` redirects to Stripe Checkout.
- [ ] After successful Stripe Checkout, returning to the app shows the Pro badge and
      updated billing settings.
- [ ] Clicking "Manage subscription" on `/settings/billing` (pro user) redirects to
      Stripe Customer Portal.
- [ ] A free user opening the "Create URL" form does not see the custom alias field;
      a tooltip indicates it is a Pro feature.
- [ ] A 403 `plan_required` API response triggers the upgrade modal.

## Out of Scope

- Invoice history display.
- Admin billing management UI.
- Mobile-specific layouts beyond standard responsive behaviour.
- Trial countdown UI.
