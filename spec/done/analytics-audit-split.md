# analytics-service Audit Split

> **Status:** done
> **Service(s):** analytics-service
> **Priority:** medium

## Goal

Remove the audit responsibilities from analytics-service so it has a single, clear
purpose: domain event metrics. The GET /events endpoint and its associated code are
deleted. The RabbitMQ event consumption for metrics is kept. The data retention cron job
is kept for metrics data. This should only be done after audit-service (audit-service.md)
is live and serving the audit trail.

## Background

analytics-service currently holds two conflicting roles:

1. Metrics: click counts, top URLs, platform summary (GET /analytics/\*). These queries
   aggregate domain events. Eventual consistency is acceptable. Data can be pruned after
   90 days.

2. Audit log: raw event trail exposed via GET /events. Users see their own events;
   admins see all. This is a compliance concern. Data must not be pruned.

These roles have incompatible retention requirements. The outbox + audit-service
initiative (Parts 1-3 of the audit trail roadmap) moves the audit log to a dedicated
service. This spec removes the redundant audit view from analytics-service.

Prerequisite: audit-service must be deployed and receiving events before this spec is
implemented. If /events is removed before audit-service is live, the audit trail has
a gap.

## Requirements

### Functional

- **R1:** The `GET /events` route, `events.routes.js`, and `events.controller.js` files
  shall be removed from analytics-service.

- **R2:** The `routes/index.js` file shall no longer mount the events routes.

- **R3:** The RabbitMQ event consumption (`consumeEvents` in index.js) shall be kept.
  analytics-service continues to consume domain events from the `events` queue for
  metrics purposes.

- **R4:** The data retention cron job (`cleanup.service.js`, `EVENT_RETENTION_DAYS`)
  shall be kept. Pruning old metrics data after 90 days is acceptable.

- **R5:** The event consumption and storage logic (`event.service.js`, the Event Prisma
  model, and all existing analytics endpoints) shall be unchanged.

- **R6:** The `docs/openapi.yaml` GET /events entry under the `analytics` tag shall be
  removed.

### Non-Functional

- **R7:** No new packages or infrastructure changes are required.

- **R8:** Existing analytics endpoint tests shall continue to pass.

- **R9:** The removed GET /events tests shall be deleted (they test a removed feature,
  not broken behaviour).

## Acceptance Criteria

- [ ] GET /events returns 404 after deployment.

- [ ] GET /analytics/urls/:shortId/clicks, GET /analytics/summary, and
      GET /analytics/top-urls continue to return correct responses.

- [ ] analytics-service continues to consume events from RabbitMQ and store them in
      the Event table (verified by checking the Event table row count increases after
      a url_clicked event).

- [ ] The data retention cron job still runs daily and deletes events older than
      EVENT_RETENTION_DAYS.

- [ ] No references to GET /events remain in docs/openapi.yaml.

- [ ] All remaining analytics-service tests pass.

## Out of Scope

- Changes to the Event Prisma model or database schema.
- Changes to analytics metrics endpoints or their behaviour.
- Adding any new endpoints to analytics-service.
- Migrating historical /events data to audit-service (historical analytics data
  stays in analytics_service DB; audit-service starts fresh from its go-live date).
