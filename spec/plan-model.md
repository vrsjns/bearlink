# Plan Model and Billing Service

> **Status:** draft
> **Service(s):** billing-service (new), auth-service, shared
> **Priority:** high

## Goal

Introduce a data model for subscription plans and a dedicated billing-service that
owns plan definitions and user subscription state. This is the foundation layer that
all other premium-plan parts depend on.

## Background

BearLink currently has no concept of paid plans. All authenticated users share the
same capabilities. To gate premium features we need a persistent, authoritative
record of which plan each user is on, and a service responsible for managing that
state. See `spec/roadmap/premium-plans-roadmap.md` for the full initiative.

An ADR must be written alongside this spec (see "ADR required" below).

## Requirements

### Functional

- **R1:** The system shall define three plan levels: `free`, `pro`, `enterprise`
  (enterprise is reserved; not purchasable via self-serve at this stage).
- **R2:** The billing-service shall expose a REST API to read a user's current plan
  and subscription status.
- **R3:** The billing-service shall store one subscription record per user, with
  fields: `userId`, `plan`, `status` (`active`, `trialing`, `past_due`, `canceled`),
  `currentPeriodEnd`, `stripeCustomerId` (nullable), `stripeSubscriptionId` (nullable).
- **R4:** All existing users shall default to plan `free` / status `active` on
  migration.
- **R5:** The billing-service shall consume the `user_registered` event from the
  RabbitMQ `events` queue and create a default subscription record (`plan: free`,
  `status: active`) for the new user.
- **R6:** The `plan` field on the auth-service User model shall default to `free`
  so that a JWT can be signed immediately at first login, before billing-service has
  had time to process the `user_registered` event and publish back.
- **R7:** The billing-service shall publish a `subscription_updated` domain event
  (via RabbitMQ `events` queue) whenever a subscription record changes.
- **R8:** Auth-service shall consume `subscription_updated` and denormalize `plan`
  onto the User record so that JWT tokens can carry the plan without a round-trip.
- **R9:** The shared JWT middleware shall include `plan` in the decoded token payload
  (added when auth-service signs the token).
- **R10:** The billing-service shall expose `GET /subscriptions/me` (user-facing,
  JWT-authenticated) returning plan, status, and period end.

### Non-Functional

- **R11:** The billing-service database shall be isolated (`billing_service`), consistent
  with the per-service database pattern (see `docs/adr/003`).
- **R12:** The service shall implement graceful shutdown on SIGTERM/SIGINT, consistent
  with other services.
- **R13:** The billing-service shall start without Stripe credentials — Stripe
  integration is added in a later part (`spec/stripe-integration.md`).

## Acceptance Criteria

- [ ] When a `user_registered` event is consumed, billing-service creates a subscription record with `plan: "free"` and `status: "active"` for that user.
- [ ] A newly registered user can log in immediately and receive a JWT with `plan: "free"` before billing-service has processed the `user_registered` event.
- [ ] `GET /subscriptions/me` returns `{ plan: "free", status: "active", currentPeriodEnd: null }` for a new user.
- [ ] After a subscription record is updated, a `subscription_updated` event appears on the `events` queue.
- [ ] Auth-service signs a JWT that contains `plan: "free"` for a free user and `plan: "pro"` for a pro user.
- [ ] `req.user.plan` is available in all services after JWT verification.
- [ ] All existing users are migrated to plan `free` / status `active` with no data loss.
- [ ] billing-service starts and passes health checks with no Stripe env vars set.

## Out of Scope

- Stripe payment integration (Part 3).
- Feature enforcement / gating (Part 2).
- UI (Part 4).
- Team/org subscriptions.
- Proration or billing cycle management.

## ADR required

Write `docs/adr/004-billing-service-architecture.md` covering:

- New billing-service vs. extending auth-service.
- Plan stored in billing DB + denormalized to JWT vs. real-time lookup.
- Event-driven sync (billing-service --> auth-service) vs. direct DB read.
