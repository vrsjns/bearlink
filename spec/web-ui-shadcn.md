# web-ui: Migrate to shadcn/ui component library

> **Status:** draft
> **Service(s):** web-ui
> **Priority:** medium

## Goal

Replace the hand-rolled Tailwind class strings scattered across every page with
shadcn/ui components, making the UI consistent, accessible, and easier to extend
without touching business logic.

## Background

The web-ui uses Tailwind CSS but has no component abstractions. Every page
repeats the same long className strings for buttons and inputs, builds its own
alert boxes, and implements accessibility-sensitive widgets (tabs, dialogs) from
scratch without ARIA support. Errors are silently swallowed with `console.error`
and never shown to the user.

shadcn/ui is the standard component layer for Next.js + Tailwind + TypeScript
projects. It copies component source files into the project (no runtime
dependency), uses Radix UI for interactive primitives, and works with the
existing Tailwind setup without adding a second styling system.

## Requirements

### Functional

- **R1:** The project shall be initialised with shadcn/ui (`npx shadcn@latest init`)
  and a `components.json` configuration file committed to the repository.
- **R2:** The following shadcn/ui components shall be installed and used:
  `Button`, `Input`, `Label`, `Card`, `Tabs`, `Badge`, `Alert`, `AlertDialog`,
  `Table`, `Toaster` (sonner).
- **R3:** All pages shall use `Button` instead of raw `<button>` elements with
  inline Tailwind class strings.
- **R4:** All form fields shall use `Input` and `Label` instead of raw `<input>`
  and `<label>` elements with inline Tailwind class strings.
- **R5:** All page card containers (white rounded boxes) shall use `Card`,
  `CardHeader`, and `CardContent`.
- **R6:** The profile page tabs shall use the shadcn/ui `Tabs` component
  (backed by Radix) instead of the hand-rolled tab buttons.
- **R7:** Success and error messages on the profile page shall use the `Alert`
  component instead of custom-styled divs.
- **R8:** The delete action on the manage page shall be confirmed via an
  `AlertDialog` before the request is sent. Currently deletion is immediate
  with no confirmation.
- **R9:** API errors that are currently silently logged with `console.error`
  on the home, login, register, and manage pages shall be surfaced to the user
  via a toast notification using `Toaster` (sonner).
- **R10:** URL tags on the manage page shall use `Badge` instead of custom
  inline-styled spans.
- **R11:** The URL table on the manage page shall use the shadcn/ui `Table`
  component (`Table`, `TableHeader`, `TableRow`, `TableHead`, `TableBody`,
  `TableCell`).

### Non-Functional

- **R12:** No business logic, API calls, routing, or state management shall
  change as part of this migration — only the rendering layer.
- **R13:** The existing test suite shall remain green after the migration.
- **R14:** The Docker image build for web-ui shall continue to succeed.

## Acceptance Criteria

- [ ] `components.json` exists at `web-ui/components.json` and `npx shadcn@latest add` works without re-running init.
- [ ] `src/components/ui/` contains all installed shadcn component files.
- [ ] No raw `<button className="bg-blue-...">` or `<input className="shadow appearance-none...">` patterns remain in any page.
- [ ] The profile page tabs are keyboard-navigable and have correct ARIA roles.
- [ ] Clicking Delete on the manage page opens a confirmation dialog; the URL is only deleted after confirmation.
- [ ] Submitting the login form with wrong credentials shows a toast error instead of a silent console.error.
- [ ] Submitting the register form with a duplicate email shows a toast error.
- [ ] Shortening a URL that returns an API error shows a toast error on the home page.
- [ ] All existing tests pass (`cd web-ui && npm test`).

## Out of Scope

- Changing any API service or backend code.
- Adding new pages or features beyond what exists today.
- Dark mode support (Tailwind dark mode variables are scaffolded by shadcn init but no dark mode switching UI is added).
- Replacing the NavBar with a shadcn navigation component — the navbar is simple enough to stay as-is.
- Adding form validation libraries (e.g. react-hook-form, zod) — validation logic already exists in the pages.
