# Stripe Integration

> **Status:** approved
> **Service(s):** billing-service
> **Priority:** high

## Goal

Connect billing-service to Stripe so users can subscribe to the Pro plan via a
hosted Checkout page, manage their subscription via the Customer Portal, and have
subscription state automatically kept in sync via webhooks.

## Background

After `spec/plan-model.md`, billing-service stores subscription state but has no
payment processor. This spec adds Stripe as the payment backend. No card data ever
touches BearLink servers -- all payment UI is hosted by Stripe.

## Requirements

### Functional

- **R1:** billing-service shall expose `POST /billing/checkout` (JWT-authenticated)
  that creates a Stripe Checkout Session for the Pro plan and returns `{ url }` for
  client-side redirect.
- **R2:** billing-service shall expose `POST /billing/portal` (JWT-authenticated)
  that creates a Stripe Customer Portal Session and returns `{ url }`, allowing
  users to manage / cancel their subscription.
- **R3:** billing-service shall expose `POST /billing/webhook` (public, Stripe
  signature-verified) to handle the following Stripe events:
  - `checkout.session.completed` -- activate subscription, update `stripeCustomerId`
    and `stripeSubscriptionId`
  - `invoice.payment_succeeded` -- extend `currentPeriodEnd`
  - `invoice.payment_failed` -- set status to `past_due`
  - `customer.subscription.deleted` -- set status to `canceled`, revert plan to `free`
  - `customer.subscription.updated` -- sync plan and status on plan changes
- **R4:** Each handled Stripe event shall trigger a `subscription_updated` event on
  the RabbitMQ `events` queue, consistent with R5 of `spec/plan-model.md`.
- **R5:** Stripe webhook signature shall be verified using `STRIPE_WEBHOOK_SECRET`
  before any processing; unverified requests return 400.
- **R6:** `STRIPE_SECRET_KEY`, `STRIPE_PRICE_ID_PRO`, and `STRIPE_WEBHOOK_SECRET`
  shall be required env vars when Stripe features are enabled; the service should
  log a clear error and exit if missing.
- **R7:** The checkout success URL and cancel URL shall be configurable via
  `FRONTEND_URL` env var.

### Non-Functional

- **R8:** Webhook handling shall be idempotent -- re-delivery of the same Stripe
  event shall not create duplicate records or events.
- **R9:** The raw request body must be preserved for Stripe signature verification
  (do not parse JSON before verification middleware).
- **R10:** Stripe SDK calls shall be wrapped in try/catch; errors return 500 without
  leaking Stripe internals.

## Acceptance Criteria

- [ ] `POST /billing/checkout` returns a Stripe Checkout URL for an authenticated
      free user; redirects an existing pro user to the portal instead.
- [ ] After completing a Stripe test-mode checkout, the user's subscription record
      updates to `plan: "pro", status: "active"` and a `subscription_updated` event is
      published.
- [ ] `POST /billing/webhook` with an invalid Stripe signature returns 400.
- [ ] Re-delivering the same `checkout.session.completed` event does not duplicate
      subscription records.
- [ ] After a simulated `customer.subscription.deleted` event, the user's plan
      reverts to `free`.
- [ ] `POST /billing/portal` returns a Stripe Customer Portal URL for a pro user.

## Out of Scope

- Annual billing / multiple price tiers (only Pro monthly in scope).
- Proration on plan changes.
- Invoice PDF generation or download.
- Trial periods.
- Refund handling.
- Enterprise plan checkout (manual sales process).
